---
title: "Flake composition model: hosts, profiles, and overlays"
description: "How descriptor attrsets, builder functions, and four distinct layers combine to produce every NixOS, nix-darwin, and Home Manager output in the flake."
---

The flake is not a collection of manually assembled system closures — it is a composition engine. Four descriptor attrsets (`users`, `hosts`, `homeProfiles`, `systemProfiles`) declare the available building blocks, three output maps (`nixosOutputDefs`, `darwinOutputDefs`, `homeOutputDefs`) select which blocks each output uses, and three builder functions (`mkOutput`, `mkDarwinOutput`, `mkHomeOutput`) turn those selections into the final flake outputs. Adding a new machine or user is a selection task, not a refactor.

## Four-layer model

Every output is the product of exactly four layers stacked in order:

| Layer | Location | Responsibility |
|---|---|---|
| **Hosts** | `hosts/<name>/` | Hardware, storage, networking, users, service-local values |
| **System profiles** | `modules/system/` | NixOS module sets: TTY baseline, compositor, gaming, Flatpak |
| **Home profiles** | `modules/home/` | Home Manager module sets: CLI tools, desktop packages, dotfiles |
| **Shared local options** | `modules/shared/options.nix` | `my.*` namespace for repo-local cross-layer values |

`flake.nix` imports `parts/lib.nix`, which defines the descriptor attrsets and builder functions and exposes them to the other parts modules through `_module.args.flakeLib`. This keeps each platform's output definitions in its own file (`parts/nixos.nix`, `parts/darwin.nix`, `parts/home.nix`) while sharing all the core logic.

## Descriptor attrsets

### `users`

Maps a user name to a Home Manager entry module. The module is responsible for shell, editor, SSH config, theme baseline, and platform-specific paths.

```nix
users = {
  gars   = { module = self + /modules/home/users/gars; };
  htmlgxn = { module = self + /modules/home/users/htmlgxn.nix; };
};
```

### `hosts`

Maps a host name to its system, entry module, extra system modules, and optional host-level Home Manager modules. `extraSystemModules` is the correct place for host-specific services (Jellyfin, containers, Soft Serve) that are not part of any named profile.

```nix
hosts = {
  boreal = {
    system = "x86_64-linux";
    module = self + /hosts/boreal/configuration.nix;
    extraSystemModules = [
      (self + /modules/system/jellyfin.nix)
      (self + /modules/system/containers.nix)
    ];
    hostHomeModules = [
      (self + /hosts/boreal/home.nix)
      (self + /modules/home/packages)
    ];
  };

  rpi4 = {
    system = "aarch64-linux";
    module = self + /hosts/rpi4/configuration.nix;
    extraSystemModules = [ nixos-hardware.nixosModules.raspberry-pi-4 ];
    hostHomeModules = [ (self + /hosts/rpi4/home.nix) ];
  };
};
```

### `homeProfiles`

Maps a profile name to the list of Home Manager modules it activates. Shared modules (`cli-base-apps`, `containers`, `fastfetch`, `nixvim`) are always included and are not repeated here — profiles layer on top of that baseline.

```nix
homeProfiles = {
  cli      = [];
  gui      = [ (self + /modules/home/gui-base-apps.nix) ];
  sway     = [ (self + /modules/home/sway.nix)
               (self + /modules/home/sway-apps.nix) ];
  sway-full = [
    (self + /modules/home/sway.nix)
    (self + /modules/home/sway-apps.nix)
    (self + /modules/home/gui-extra-apps.nix)
    (self + /modules/home/flatpak.nix)
    (self + /modules/home/gaming.nix)
  ];
};
```

### `systemProfiles`

Maps a profile name to the list of NixOS modules it activates. Shared system modules (Lix, `my.*` options, timezone, locale, Nix settings) are always included and are not part of any named profile.

```nix
systemProfiles = {
  tty      = [ (self + /modules/system/cli.nix) ];
  sway     = [ (self + /modules/system/cli.nix)
               (self + /modules/system/sway.nix) ];
  sway-full = [
    (self + /modules/system/cli.nix)
    (self + /modules/system/sway.nix)
    (self + /modules/system/flatpak.nix)
    (self + /modules/system/gaming.nix)
  ];
};
```

### Output maps: `nixosOutputDefs`, `darwinOutputDefs`, `homeOutputDefs`

Each output map is an attrset of output name → descriptor. A descriptor selects a host (NixOS), or supplies `system` directly (Darwin/HM), plus a user, profiles, and optional overlays.

<Tabs>
  <Tab title="NixOS">
    ```nix
    # parts/nixos.nix
    nixosOutputDefs = {
      boreal = {
        hostName      = "boreal";
        userName      = "gars";
        systemProfile = "sway-full";
        homeProfile   = "sway-full";
        homeOverlays  = ["ai"];
        nixpkgsOverlays = [ borealOverlay ];
      };

      rpi4-sway = {
        hostName      = "rpi4";
        userName      = "gars";
        systemProfile = "sway";
        homeProfile   = "sway";
        homeOverlays  = ["ai"];
      };
    };

    flake.nixosConfigurations = builtins.mapAttrs (_: mkOutput) nixosOutputDefs;
    ```
  </Tab>
  <Tab title="nix-darwin">
    ```nix
    # parts/darwin.nix
    darwinOutputDefs = {
      macbook = {
        userName         = "htmlgxn";
        homeProfile      = "gui";
        system           = "aarch64-darwin";
        hostHomeModules  = [ (self + /hosts/macbook/home.nix) ];
        homeOverlays     = ["ai"];

    };

    flake.darwinConfigurations = builtins.mapAttrs (_: mkDarwinOutput) darwinOutputDefs;
    ```
  </Tab>

## Builder functions

All three builders live in `parts/lib.nix` and are exposed to the other parts modules via `_module.args.flakeLib`.

### `mkOutput` — NixOS

Assembles a `nixpkgs.lib.nixosSystem` from:

1. `sharedSystemModules` (Lix, `my.*` options, global locale/Nix settings)
2. A nixpkgs config inline module (platform, `allowUnfree`, optional nixpkgs overlays)
3. The host entry module (`host.module`)
4. The selected `systemProfile` module list
5. `host.extraSystemModules` (host-specific services)
6. The `home-manager.nixosModules.home-manager` module
7. An inline Home Manager configuration built by `mkHomeModule`

```nix
mkOutput = {
  hostName, userName, systemProfile, homeProfile,
  homeOverlays ? [], nixpkgsOverlays ? [],
}: let
  host = hosts.${hostName};
in nixpkgs.lib.nixosSystem {
  modules =
    sharedSystemModules
    ++ [ (_: { nixpkgs = {
                 hostPlatform = host.system;
                 config.allowUnfree = true;
                 overlays = nixpkgsOverlays; }; }) ]
    ++ [ host.module ]
    ++ systemProfiles.${systemProfile}
    ++ host.extraSystemModules
    ++ [ home-manager.nixosModules.home-manager
         (mkHomeModule { inherit userName homeProfile homeOverlays;
                         hostHomeModules = host.hostHomeModules or []; }) ];
};
```

### `mkDarwinOutput` — nix-darwin

Assembles a `nix-darwin.lib.darwinSystem`. The macbook host configuration is hardcoded inside the builder (there is only one Darwin target), so the descriptor supplies the user, home profile, system architecture, optional host-level HM modules, and overlay groups.

```nix
mkDarwinOutput = {
  userName, homeProfile, system,
  hostHomeModules ? [], homeOverlays ? [],
}:
  nix-darwin.lib.darwinSystem {
    inherit system;
    specialArgs = { inherit inputs; };
    modules = [
      inputs.lix-module.nixosModules.default
      (self + /hosts/macbook/configuration.nix)
      home-manager.darwinModules.home-manager
      { nixpkgs.config.allowUnfree = true;
        home-manager.useGlobalPkgs = true;
        home-manager.useUserPackages = true;
        home-manager.extraSpecialArgs = { inherit inputs; };
        home-manager.users.${userName} = {
          imports = mkHomeImports { inherit userName homeProfile
                                           hostHomeModules homeOverlays; };
          home.homeDirectory = nixpkgs.lib.mkForce "/Users/${userName}";
        }; }
    ];
  };
```

### `mkHomeOutput` — standalone Home Manager

Assembles a `home-manager.lib.homeManagerConfiguration` for targets that run NixOS, Fedora, or other non-Darwin Linux without a shared system closure (Jetson Orin Nano, Fedora-on-Mac).

```nix
mkHomeOutput = {
  userName, homeProfile, system,
  hostHomeModules ? [], homeOverlays ? [],
}: let
  pkgs = nixpkgs.legacyPackages.${system};
in home-manager.lib.homeManagerConfiguration {
  inherit pkgs;
  extraSpecialArgs = { inherit inputs; };
  modules =
    mkHomeImports { inherit userName homeProfile hostHomeModules homeOverlays; }
    ++ [ { nixpkgs.config.allowUnfree = true;
           home.username = userName;
           home.homeDirectory = "/home/${userName}";
           home.stateVersion = "26.05"; } ];
};
```

## Output assembly order

Every output is understandable as a strict stack. For NixOS, the precedence from lowest to highest is:

<Steps>
  <Step title="Shared defaults">
    `sharedSystemModules` applies global Lix, locale, Nix settings, and `my.*` option declarations to every output.
  </Step>
  <Step title="Host">
    `host.module` and `host.extraSystemModules` add machine-specific hardware, storage, networking, and services.
  </Step>
  <Step title="System profile">
    The selected `systemProfile` (e.g. `sway-full`) layers in the NixOS module set for that desktop or TTY configuration.
  </Step>
  <Step title="User and home profile">
    `mkHomeModule` wires `home-manager` with the selected user module, home profile modules, and host-level HM modules.
  </Step>
  <Step title="Overlay groups">
    Named `homeOverlays` (e.g. `["ai"]`) append extra Home Manager modules. `nixpkgsOverlays` modify the nixpkgs package set (e.g. Brave Nightly on boreal).
  </Step>
</Steps>

Keep each selection visible at the output descriptor. If an output needs AI tooling or host-specific HM modules, that should be obvious from the descriptor — not hidden inside an unrelated profile or module.

## Ownership rules

Use these rules to decide where new code belongs:

<AccordionGroup>
  <Accordion title="Good fits for hosts/">
    - Filesystem layout, swap, zram
    - Firewall ports specific to one machine
    - Service-local paths (Jellyfin data dir, Borg repo path)
    - GPU quirks tied to one machine (AMD graphics, `enable32Bit`)
    - Host-local SSH targets or `my.*` overrides
  </Accordion>
  <Accordion title="Good fits for modules/system/">
    - Reusable service logic (Jellyfin, containers, Soft Serve)
    - Compositor plumbing (Sway system services)
    - Host-agnostic package groups
    - Shared container runtime behavior
  </Accordion>
  <Accordion title="Good fits for modules/home/">
    - Shell aliases and helper functions
    - User-facing package groups
    - Application config and dotfile wiring
    - Theme selectors
  </Accordion>
  <Accordion title="Poor fits for host modules">
    - General CLI package sets — put these in a home profile or `modules/home/`
    - Reusable desktop package sets — these belong in a named home or system profile
    - Service logic that can be parameterized via `my.*` — extract to `modules/system/` and supply values from the host
  </Accordion>
</AccordionGroup>
