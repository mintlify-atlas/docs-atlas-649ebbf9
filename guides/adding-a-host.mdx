---
title: "Add a new host to the flake configuration"
description: "Create a host directory, register it in parts/lib.nix, attach system and home profile modules, and add output entries in parts/nixos.nix."
---

Adding a host means registering a new machine so the flake can produce a `nixosConfiguration` (or `darwinConfiguration`) for it. The process has three distinct parts: creating the host files, registering the host descriptor in `parts/lib.nix`, and adding one or more output entries in the relevant `parts/` file. Keeping these layers separate is what lets multiple outputs share the same host while selecting different profiles.

<Note>
Never edit `hosts/*/hardware-configuration.nix` by hand. That file is generated by `nixos-generate-config` on the target machine and is intentionally excluded from formatting and automation. Copy it from the target host and commit it as-is.
</Note>

## Steps

<Steps>
  <Step title="Create the host directory and configuration entrypoint">
    Create `hosts/<hostname>/configuration.nix`. For a simple host this file can contain all the host-local settings directly. For a non-trivial host, keep it thin — a minimal import list — and split host-local concerns into focused files under the same directory.

    Boreal is the canonical example of a split host:

    ```
    hosts/boreal/
      configuration.nix   ← thin import list
      base.nix            ← bootloader, kernel, binfmt
      graphics.nix        ← AMD GPU, 32-bit graphics
      storage.nix         ← mounts, zram, tmpfiles
      networking.nix      ← firewall, open ports
      users.nix           ← NixOS user accounts
      services.nix        ← service-local my.* values
      home.nix            ← host-local HM overrides
    ```

    The `configuration.nix` entrypoint for boreal is just:

    ```nix
    # hosts/boreal/configuration.nix
    {...}: {
      imports = [
        ./hardware-configuration.nix
        ./base.nix
        ./graphics.nix
        ./storage.nix
        ./networking.nix
        ./users.nix
        ./services.nix
      ];

      system.stateVersion = "25.11";
    }
    ```

    A simpler single-file host like `nixos-vm` puts everything directly in `configuration.nix`.
  </Step>

  <Step title="Register the host descriptor in parts/lib.nix">
    Open `parts/lib.nix` and add an entry to the `hosts` attrset. Each descriptor provides the system architecture, the path to the host module, any shared modules that are still host-scoped, and any host-local Home Manager modules.

    ```nix
    hosts = {
      # ... existing hosts ...

      myhostname = {
        system = "x86_64-linux";  # or "aarch64-linux"
        module = self + /hosts/myhostname/configuration.nix;
        extraSystemModules = [
          # Add shared NixOS modules that are host-scoped by values or hardware.
          # Example: (self + /modules/system/jellyfin.nix)
          # Leave empty if no shared modules need host-specific values.
        ];
        hostHomeModules = [
          # Add host-local Home Manager modules here.
          # Example: (self + /hosts/myhostname/home.nix)
        ];
      };
    };
    ```

    Use `extraSystemModules` only for shared modules that require host-specific values (like `my.jellyfin.*`) or hardware-specific imports. Use `hostHomeModules` for Home Manager configuration that belongs to the machine rather than the user — terminal overrides, self-referential SSH targets, and similar host-scoped knobs.
  </Step>

  <Step title="Add the output definition">
    Open `parts/nixos.nix` (or `parts/darwin.nix` for a nix-darwin host) and add an entry to the relevant output map. Each output selects a host, a user, a system profile, a home profile, and any explicit home overlay groups.

    ```nix
    nixosOutputDefs = {
      # ... existing outputs ...

      myhostname-tty = {
        hostName = "myhostname";
        userName = "gars";          # must match a key in the users attrset
        systemProfile = "tty";      # tty | sway | sway-full
        homeProfile = "cli";        # cli | gui | sway | sway-full | sway-config
        homeOverlays = [];          # [] or ["ai"]
      };
    };
    ```

    If you want a second output for the same host with a different profile — for example a full Sway desktop alongside a TTY-only variant — add a second entry with a different output key and profile selections. Both share the same host descriptor in `parts/lib.nix`.
  </Step>

  <Step title="Validate the new output">
    Run a dry-build to confirm the new output evaluates without errors:

    ```bash
    nrb myhostname-tty
    ```

    Or use the dry-run diff to see what the output would produce:

    ```bash
    nrd myhostname-tty
    ```

    Fix any evaluation errors before applying. The `ncheck-full` command will also evaluate all output families and is worth running before committing.
  </Step>
</Steps>

## System and home profiles

The profiles available are defined in `parts/lib.nix`:

**System profiles** (`systemProfile`):

| Profile    | Modules included                                      |
| ---------- | ----------------------------------------------------- |
| `tty`      | `modules/system/cli.nix`                             |
| `sway`     | `cli.nix` + `sway.nix`                               |
| `sway-full`| `cli.nix` + `sway.nix` + `flatpak.nix` + `gaming.nix` |

**Home profiles** (`homeProfile`):

| Profile      | Modules included                                            |
| ------------ | ----------------------------------------------------------- |
| `cli`        | (shared modules only)                                       |
| `gui`        | `gui-base-apps.nix`                                         |
| `sway-config`| `sway.nix` (config-only, packages set to null)              |
| `sway`       | `sway.nix` + `sway-apps.nix`                               |
| `sway-full`  | `sway.nix` + `sway-apps.nix` + `gui-extra-apps.nix` + `flatpak.nix` + `gaming.nix` |

ARM hosts (`rpi4`) use the `sway` profiles rather than `sway-full` to omit Flatpak and gaming.
